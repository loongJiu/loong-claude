# 将想法完善成方案

通过自然的协作对话，将输入的想法转化为完整的设计方案。

首先了解当前项目背景，然后逐个提问以完善想法。一旦你明白要构建什么，就展示方案并获得用户批准。

在展示方案并得到用户批准之前，不要调用任何实施技能、编写任何代码、搭建任何项目或采取任何实施行动。这条规则适用于**所有**项目。

# 反模式：「这个太简单了，不需要设计」

任何项目都应该先经过设计阶段。

无论是一个待办清单、一个只有单一功能的小工具，还是一次简单的配置修改，都不例外。

**不要**认为“简单项目”可以直接开始动手，因为实际上，真正容易造成返工和浪费的，往往正是这些看似简单的需求。因为其中隐藏的大量默认假设，通常从未被明确讨论或确认。

对于非常简单的项目，设计文档不一定要复杂，几句话的说明可能就已经足够。

但无论项目大小，你都必须：

1. 先明确设计思路
2. 向用户展示方案
3. 在获得确认后再开始实施

这样做的目的，不是增加流程负担，而是避免因为理解偏差而产生后续的大量无效工作。

# Checklist

你必须为以下每一项创建独立任务，并严格按照顺序完成：

1. **探索项目上下文**
   * 检查相关代码文件、文档以及最近的提交记录
   * 先理解项目现状，再进入后续流程

2. **提出澄清问题**
   * 一次只问一个问题
   * 逐步理解：
     * 用户目标
     * 约束条件
     * 成功标准
   * 不要一次性抛出大量问题

3. **提出 2～3 种实现方案**
   * 分析每种方案的优缺点与取舍（trade-offs）
   * 明确给出推荐方案及推荐理由

4. **展示设计方案**
   * 按复杂度拆分为多个部分逐步展示
   * 每展示一个部分，都需要获得用户确认后再继续下一部分

5. **编写设计文档**
   * 将设计保存到：
     ```text
     .claude/loong-claude/specs/YYYY-MM-DD-<topic>-design.md
     ```
   * 此文档不需要commit到git仓库

6. **进行设计文档自检**
   * 快速检查：
     * 是否存在占位内容
     * 是否有逻辑矛盾
     * 是否存在歧义
     * 是否超出范围
   * 具体检查标准见下方说明

7. **让用户审核设计文档**
   * 明确要求用户先阅读并确认 spec 文件
   * 在用户确认之前，不进入实现阶段

8. **进入实现阶段**
   * 使用loong-claude:writing-plans skill生成正式的 implementation plan（实施计划）

# 核心流程

## 1.需求洞察与规模评估

在动手之前，首要任务是确立全局观并定义边界。

现状调研： 优先审视当前项目的既有状态，包括文件结构、文档说明及近期提交记录。

规模预判： 评估需求复杂度。若需求包含多个独立的子系统（如“同时构建聊天、存储、计费和分析平台”），必须立即指出。拒绝在未解耦的大型项目细节上浪费时间。

任务拆解： 若项目过于庞大，应引导用户将其拆分为相互关联但逻辑独立的子项目。确立构建优先级，并针对首个子项目开启“设计-计划-实施”循环。

深度精炼： 针对粒度合适的任务，通过单次提问的方式逐步完善想法。优先使用多选题以提高效率，核心目标是明确项目意图、约束条件及交付标准

## 2. 技术方案探索

在进入细节设计前，提供多样化的路径选择。

方案对比： 提出 2-3 种具有不同权衡（Trade-offs）的设计方案。

决策建议： 以对话形式展示方案，并明确给出推荐路径及其背后的逻辑支撑。

## 3. 系统设计呈现

设计方案应由浅入深，确保双方共识。

动态详略： 根据模块复杂度调整篇幅。简单部分简单描述，复杂逻辑则需详尽描述（200-300字）。

迭代确认： 采取“分段呈现、即时确认”的策略，在涵盖系统架构、设计模式、组件、页面、数据流、异常处理的同时，确保每一步都经过用户认可。

如果有内容不清楚或有歧义，必须立即与用户确认。

当你认为设计完成时，就向用户展示方案。

## 4. 隔离性与清晰度

设计的核心目标是降低认知负担。

单一职责： 将系统拆分为功能单一的微小单元，通过定义良好的类型接口进行通信，确保每个单元可独立理解、独立测试。

黑盒原则： 单元应做到“无须阅读内部实现即可知晓用途”。如果你无法在不破坏调用方的前提下修改内部逻辑，说明边界划分仍需优化。

粒度控制： 文件过大通常是功能过载的信号。更小、边界更清晰的单元有助于模型进行更精准的推理和代码生成。

## 5. 既有代码库协作逻辑

在尊重现状的基础上进行渐进式改进。

模式遵循： 在提议任何变更前，深入理解并遵循现有的代码结构与风格。

针对性优化： 若既有代码存在影响当前任务的问题（如文件臃肿、边界不清、职责纠缠），应将针对性的重构作为设计方案的一部分，并向用户说明优化原因。

保持专注： 拒绝与当前目标无关的大规模重构，确保所有改动均直接服务于当前任务。

# 设计完成后的流程

## 1. 文档沉淀

在设计获得认可后，需将其正式文档化：

路径归档： 将验证后的设计方案（技术规范/Spec）写入 .claude/loong-claude/specs/YYYY-MM-DD-<topic>-design.md。

注：若用户指定了存储位置，则优先遵循用户偏好。

写作风格： 运用“清晰简练”的写作技巧，确保内容易读且无冗余。

版本控制： 不要将设计文档提交到 Git 仓库，因为设计文档不是代码，而是文档。

## 2. 技术规范自审 (Spec Self-Review)

文档编写完成后，需以审查者的视角进行复盘，确保方案质量：

清理占位符： 检查是否存在“待定（TBD）”、“待办（TODO）”、不完整章节或模糊的要求，并将其完善。

内部一致性： 确认各章节是否冲突。架构设计必须与功能描述完全匹配。

规模复核： 确认当前范围是否足够聚焦，足以支撑单次的实施计划；若依然过于庞大，需进一步拆解。

消除歧义： 检查需求是否存在多种解读。若有歧义，需要立即向用户确认。

原则： 发现问题后直接就地修正。无需反复自审，修正后即刻进入下一阶段。

## 3. 用户确认环节 (User Review Gate)

自审通过后，必须设置“用户确认点”，在继续后续工作前征得许可：

固定话术： “技术规范已编写并提交至 <路径>。请您审阅，在开始制定实施计划之前，如果您有任何需要修改的地方，请随时告知。”

操作规程： 等待用户反馈。若用户提出修改意见，则更新文档并重新运行自审流程。只有在用户明确批准后，方可推进。

## 4. 实施阶段 (Implementation)

一旦获得授权，立即进入执行环节：

启动计划： 使用loong-claude:writing-plans skill制定详细的实施计划。

# 核心原则

* **单次单问：** 每次仅提一个问题，避免信息过载。
* **选项优先：** 尽可能提供多选题，这通常比开放式提问更易于回答。
* **极致简化 (YAGNI)：** 严格执行 YAGNI 原则（You Ain't Gonna Need It），从所有设计中无情地剔除不必要的功能。
* **方案探索：** 在最终定稿前，务必提出 2-3 种可选方案。
* **增量验证：** 遵循“呈现设计 - 获得批准 - 继续推进”的循环。
* **保持灵活性：** 一旦发现逻辑不通，随时回溯并进一步澄清。

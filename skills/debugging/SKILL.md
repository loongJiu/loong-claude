# 调试

核心范式：**ReAct 循环**（观察 → 推理 → 行动 → 观察）——不凭感觉猜，系统性定位后再修复。

**铁律**：在确认根因之前，**绝不修改代码**。草率的修复只会掩盖症状，制造新隐患。

---

## 阶段一：收集症状（初始 Observe）

开始时声明："我正在使用 `debugging` skill 进行系统性调试。"

首先掌握完整上下文，缺任何一项都要主动索取：

| 信息 | 获取方式 |
|------|---------|
| 完整错误信息 / stack trace | 用户提供或读终端输出 |
| 复现路径 | 询问用户：必现还是偶现，什么操作触发 |
| 预期行为 | 询问用户：正确状态应该是什么 |
| 最近变更 | `git log --oneline -10` + `git diff HEAD~1` |

```bash
git log --oneline -10
git diff HEAD~1 -- <疑似相关文件>
```

---

## 阶段二：ReAct 调试循环

每一轮严格遵循三步，不得跳过：

### Observe（观察）

从报错的最近调用点出发，沿调用链追踪，读取相关代码、类型定义、配置：

```bash
# 读报错文件及其直接依赖
# 追溯调用链上下游
# 检查相关类型定义
# 若涉及接口：确认请求参数和响应结构
```

> 原则：每次 Observe 只追踪与当前假设直接相关的文件，不要漫无目的地"把整个目录读一遍"。

### Reason（推理）

基于本轮观察，列出所有假设，**每条假设必须有对应证据**：

```
假设列表（按可能性降序）：
1. [最可能] <原因描述> — 证据：<具体代码行 / 日志内容>
2. [次可能] <原因描述> — 证据：<具体代码行 / 日志内容>
3. [备选]   <原因描述> — 证据：<具体代码行 / 日志内容>
```

没有证据支撑的假设一律降到最低优先级。

### Act（验证）

针对优先级最高的假设，选择**最小成本**的验证方式：

**读代码验证**（首选，零成本）
```typescript
// 直接检查相关逻辑是否与假设吻合
```

**加临时日志**（低成本）
```typescript
console.log('[debug]', { 变量名: 变量值 });
```

**运行命令验证**
```bash
npx tsc --noEmit                              # 类型错误
npm test -- --testNamePattern="描述关键词"    # 特定测试用例
```

**隔离复现**：最小化复现场景，排除无关变量干扰。

### 循环判断

| 结果 | 下一步 |
|------|--------|
| 假设成立 | 进入阶段三（修复） |
| 假设不成立 | 标记为「已排除」，从假设列表移除，进入下一轮 |
| 出现新线索 | 将新线索纳入 Observe，更新假设列表，重新 Reason |

**退出条件**：能用一句话清晰描述根因，且有具体代码位置或日志作为直接证据。

---

## 阶段三：修复

定位根因后，先向用户说明再动手：

> "根因是 [一句话描述]，位于 [文件:行号]。我的修复方案是 [具体改动]。"

修复原则：
- **最小侵入**：只改与根因直接相关的代码
- **不顺手重构**：发现其他问题记录下来，不在此处处理
- **保持接口稳定**：优先不破坏现有函数签名和组件 Props

修复后立即验证：
```bash
# 重新触发之前的复现路径
# 运行相关测试
# npx tsc --noEmit 确认无新增类型错误
```

---

## 阶段四：防复发小结

修复完成后输出：

```
根因：[一句话]
修复：[改了什么，为什么这样改]
不会复发的原因：[机制说明]
建议补充测试：[是/否，如果是，描述测试场景]
顺手发现的其他问题（不处理，仅记录）：[列表或"无"]
```

---

## 禁止的反模式

- ❌ 看到报错直接改代码，不先理解原因
- ❌ 同时修改多处，导致无法判断哪个改动生效
- ❌ 用 `// @ts-ignore` 或 `as any` 绕过类型错误而不修根因
- ❌ "试试这个看看能不能行"——每次 Act 必须有对应假设
- ❌ 反复修改同一处代码超过 3 次仍未解决——此时应退回 Reason 阶段重新审视假设列表

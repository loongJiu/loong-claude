# 编写实施计划 (Writing Plans)

## 核心理念
将复杂的技术规范拆解为详尽的**微型（Bite-sized）任务**实施计划。

* **基本假设**：假设执行者是资深开发者，但对当前代码库、业务领域及测试规范**完全不了解**。计划必须包含执行所需的一切细节（修改的文件、具体代码、参考文档）。
* **核心原则**：DRY（不重复）、YAGNI（按需实现）、TDD（测试驱动开发）。
* **启动规范**：
  1. 开始时必须声明：“我正在使用 `writing-plans` skill 创建实施计划。”
  2. 默认保存路径：`.claude/loong-claude/plans/YYYY-MM-DD-<feature-name>.md`。

## 1. 范围与结构定义 (Scope & Structure)
在定义具体任务前，必须明确任务边界并映射文件职责，锁定解耦决策：

* **规模控制**：若技术规范涵盖多个独立子系统，必须建议用户将其拆分为多个独立的实施计划。
* **职责划分**：文件边界需清晰，每个文件仅承担单一职责，优先保持小而精。按业务职责而非技术层（如 MVC）聚合需同步修改的文件。
* **尊重既有规范**：遵循当前代码库的架构习惯。若需修改的文件已严重臃肿，可将拆分重构逻辑纳入计划，但**严禁**单方面进行大规模重构。

## 2. 计划文档规范 (Plan Specification)
每个实施计划**必须**包含统一的头部声明和标准化的任务结构。

### 文档头部 (Header)
# [功能名称] 实施计划

**目标：** [用一句话描述核心构建目标]
**架构/设计模式：** [用 2-3 句话描述技术方案]
**技术栈：** [核心技术/库]

### 任务颗粒度与结构 (Task Structure)

每个任务必须是一个**原子操作（预计耗时 2-10 分钟）**，并严格遵循 TDD 循环：
*步骤：1. 编写失败的测试 -> 2. 确认测试失败 -> 3. 编写最简代码通过测试 -> 4. 确认测试通过。*

**每个任务项必须明确包含：**

* **涉及文件**：明确是 Create（创建）、Modify（修改）还是 Test（测试）及其具体路径。
* **具体步骤**：提供实际可运行的测试代码、执行命令和预期输出。

## 3. 禁飞区：严禁占位符 (No Placeholders)

任务步骤必须提供工程师可直接执行的实际内容。出现以下任一情况即视为**计划失效**，绝对禁止：

* 使用 "TBD"、"TODO"、"稍后实现" 或 "填充细节"。
* 仅有概念描述而无具体方案（如要求“添加错误处理”，却不给具体代码）。
* 要求“编写测试”却不提供实际测试代码。
* 使用“同任务 N”等省略表述（工程师可能会乱序阅读，必须写出完整代码）。
* 引用任何未在当前或前置任务中明确定义的类型、函数或方法。

## 4. 自审清单 (Self-Review)

完成计划后，必须立即对照技术规范（Spec）进行校验。发现问题**原地修正**，修正后直接推进，无需重复自审：

1. **规范覆盖率**：确保 Spec 中的每项需求都有对应的落地任务。
2. **消除占位符**：全局扫描并修复上述所有“计划失效”项。
3. **类型一致性**：校验所有任务间的函数签名、属性命名等上下文是否完全一致。

## 5. 执行交接 (Execution Handoff)

保存计划后，向用户提供以下执行选项并等待选择：

“计划已完成并保存至 `.claude/loong-claude/plans/<filename>.md`。有两种执行方式，请选择：
> 1. **子代理驱动 (Subagent-Driven) [推荐]** —— 为每个任务指派独立子代理，任务间自动评审，迭代速度快。
> 2. **内联执行 (Inline Execution)** —— 在当前会话中批量执行，并在关键节点设置检查点。”

**⚠️ 提交流程红线**：在实施过程中严禁自行 commit 代码。必须在所有任务执行完毕且获得用户确认后，调用 `loong-claude:commit` skill 进行统一代码提交。

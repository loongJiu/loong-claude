# 编写实施计划 (Writing Plans)

## 核心理念
将技术规范拆解为**单次可独立验证的最小变更**序列。

* **基本假设**：假设执行者是资深开发者，但对当前代码库、业务领域及测试规范**完全不了解**。计划必须包含执行所需的一切细节（修改的文件、具体代码、参考文档）。
* **核心原则**：DRY（不重复）、YAGNI（按需实现）、测试覆盖（策略见下方）。
* **启动规范**：
  1. 开始时必须声明："我正在使用 `writing-plans` skill 创建实施计划。"
  2. 默认保存路径：`.claude/loong-claude/plans/YYYY-MM-DD-<feature-name>.md`。

---

## Step 0：复杂度路由（每次必做，优先执行）

**在定义任何任务之前**，先判断当前规范的规模，选择执行路径：

```
判断条件（满足任意一项 → 完整计划）：
  ✦ 预计任务数 > 8
  ✦ 涉及独立模块 > 3
  ✦ 存在跨任务的共享状态、接口契约或数据迁移

否则 → 精简计划
```

| 路径 | 文档内容 | Reflexion | 执行交接 |
| --- | --- | --- | --- |
| **精简计划** | 头部 + 任务列表 | 仅第 1 轮（完整性） | 直接执行，无需选择 |
| **完整计划** | 头部 + 任务列表 + 依赖图 | 两轮（完整性 + 正确性） | 提供子代理 / 内联选项 |

路由结果必须在声明行后立即输出，例如：
> 「判断：任务数 ≤ 8，单模块 → **精简计划**」
> 「判断：涉及 4 个模块，有接口契约变更 → **完整计划**」

---

## §1. 范围与结构定义 (Scope & Structure)

在定义具体任务前，必须明确任务边界并映射文件职责：

* **规模控制**：若技术规范涵盖多个独立子系统，且预计任务数 > 8 或涉及模块 > 3，**必须**建议用户将其拆分为多个独立的实施计划，而不是在一个计划里堆叠。
* **职责划分**：文件边界需清晰，每个文件仅承担单一职责，优先保持小而精。按业务职责而非技术层（如 MVC）聚合需同步修改的文件。
* **尊重既有规范**：遵循当前代码库的架构习惯。若需修改的文件已严重臃肿，可将拆分重构逻辑纳入计划，但**严禁**单方面进行大规模重构。

---

## §2. 计划文档规范 (Plan Specification)

### 文档头部 (Header)

每个实施计划**必须**以 YAML frontmatter 开头，供后续会话快速定位和检索：

```yaml
---
feature: <功能名称>
created_at: YYYY-MM-DD
complexity: simple | moderate | complex
spec_ref: .claude/loong-claude/spec/YYYY-MM-DD-<topic>-design.md
affected_modules:
  - 模块名称
task_count: N
status: draft | approved
---
```

正文头部：

```
# [功能名称] 实施计划

目标：[用一句话描述核心构建目标]
架构/设计模式：[用 2-3 句话描述技术方案]
技术栈：[核心技术/库]
```

---

### 任务原子性定义

每个任务必须是**单次可独立验证的最小变更**。判断标准（三条全部满足才算原子）：

```
① 有明确的完成信号：测试通过 / 命令输出符合预期 / 页面/接口行为可验证
② 可单独回滚：失败时撤销本任务不影响其他任务的完成状态
③ 无前向依赖：不引用本任务之后才定义的类型、函数、文件路径
```

不以时间估算（如「2-10 分钟」）定义粒度——时间对 AI 执行者是无意义的度量，可独立验证才是。

---

### 测试策略（条件触发，非全局强制）

根据任务类型选择测试方式，**不强制对所有任务走 TDD 循环**：

| 任务类型 | 推荐策略 |
| --- | --- |
| 纯函数、业务逻辑、工具类、算法 | **TDD**：先写失败测试 → 最简实现 → 通过 |
| API 接口、数据层、状态管理 | **先实现后测试**：实现完成后补集成测试 |
| UI 组件、样式、配置变更 | **冒烟验证**：运行命令确认渲染/构建通过 |
| 数据迁移、脚本、一次性任务 | **幂等验证**：确认重复执行结果一致即可 |

选择 TDD 时，步骤为：1. 编写失败测试 → 2. 确认测试失败 → 3. 编写最简实现 → 4. 确认测试通过。

---

### 每个任务项必须包含

* **涉及文件**：明确标注 Create（创建）、Modify（修改）、Test（测试）及具体路径。Import 路径必须对应已存在或将在本任务中 Create 的文件，不得引用尚未定义的路径。
* **具体步骤**：提供实际可运行的代码示例、执行命令和预期输出。
* **完成信号**：明确说明如何验证本任务已完成（测试命令 + 预期输出，或页面/接口的可观测状态）。

---

## §3. 禁飞区：严禁占位符 (No Placeholders)

出现以下任一情况即视为**计划失效**，绝对禁止：

* 使用 "TBD"、"TODO"、"稍后实现" 或 "填充细节"。
* 仅有概念描述而无具体方案（如要求"添加错误处理"，却不给具体代码）。
* 要求"编写测试"却不提供实际测试代码。
* 使用"同任务 N"等省略表述（任务必须可乱序独立阅读，写出完整代码）。
* 引用任何未在当前或前置任务中明确定义的类型、函数或方法。
* Import 路径指向当前任务之后才会创建的文件。

---

## §4. 两轮 Reflexion 自审

计划初稿完成后，强制进行两轮反思迭代。**两轮职责正交，检查对象不同，两轮都通过才能进入 §5。**

---

### 第 1 轮：完整性审查（需求 → 任务的覆盖率）

**检查维度**：Spec 里的每一项需求，在计划里是否都有对应任务？逐条比对。

```
□ 每一项功能需求都有对应的实现任务吗？
□ 每一项需求都有对应的验证方式（测试任务或完成信号）吗？
□ 有没有隐含的需求被遗漏（错误处理、边界 case、权限校验）？
□ 每个任务的涉及文件路径是明确且存在（或将被创建）的吗？
□ 有任何 TBD / TODO / 「稍后实现」吗？（有就是计划失效）
```

把所有"否"的项列出来，逐一补充对应任务。

**第 1 轮退出条件**：Spec 的每一项需求都能在计划中找到对应任务，且没有遗漏的验证方式。

---

### 第 2 轮：正确性审查（任务间接口与依赖一致性）

**检查维度**：任务内部和任务之间的实现是否自洽？

```
□ 同一个函数/接口/类型在不同任务里的签名完全一致吗？
□ 任务 N 依赖任务 M 的产出，M 排在 N 之前吗？（依赖无环，拓扑有序）
□ 测试代码引用的函数名、路径、mock 数据与实现任务中定义的完全一致吗？
□ 有没有某个任务的「完成信号」依赖一个尚未在前置任务中建立的环境或状态？
□ 精简计划：任务间是否存在隐含的共享状态，导致任务无法真正独立回滚？
```

发现问题原地修正，不需要写修正记录。

**第 2 轮退出条件**：上面 5 个问题全部确认，且没有发现新的修正项。

---

> 如果第 2 轮发现了大量问题，说明第 1 轮执行不到位（遗漏了需要定义接口的任务）。重新从第 1 轮开始。

---

## §5. 执行交接 (Execution Handoff)

### 精简计划

直接告知用户可以开始执行，无需选择执行模式：

> 「计划已完成并保存至 `.claude/loong-claude/plans/<filename>.md`。任务数较少且相互独立，可直接开始执行。」

### 完整计划

根据任务依赖结构，**先给出推荐**，再让用户确认：

**判断标准：**

```
任务间存在顺序依赖或共享上下文（如前一个任务的输出是下一个任务的输入）
  → 推荐内联执行：上下文在同一会话中传递，依赖关系可靠

任务相互独立、可并行（各任务的输入输出不交叉）
  → 推荐子代理驱动：独立指派，迭代速度更快
```

固定话术：

> 「计划已完成并保存至 `.claude/loong-claude/plans/<filename>.md`。
>
> 推荐执行方式：**[子代理驱动 / 内联执行]**，原因：[一句话说明判断依据]。
>
> 也可以选择另一种方式：
> 1. **子代理驱动** —— 为每个任务指派独立子代理，任务间自动评审，适合独立任务并行。
> 2. **内联执行** —— 在当前会话中顺序执行，在关键节点设置检查点，适合有依赖的任务链。」

---

**⚠️ 提交红线**：在实施过程中严禁自行 commit 代码。必须在所有任务执行完毕且**获得用户明确确认**后，执行统一代码提交。回复 **✓** 即视为确认。

## §6. code-review

当所有任务执行完毕且获得用户确认后，让用户选择是否调用 `loong-claude:code-review` skill 进行统一代码审查。

# Code Review

代码质量审查关卡，位于编码与提交之间：`coding → code-review → commit`。

**核心范式**：结构化审查 + LLM 可靠性分层 + 有界修复循环 + 用户决策驱动。

**铁律**：审查只产出报告和建议，**未经用户明确授权，绝不自行修改代码**。

---

## Step 1. 收集变更上下文

开始时必须声明："我正在使用 `code-review` skill 进行代码审查。"

```bash
# 变更范围
git diff --stat
git diff --cached --stat

# 具体变更内容
git diff
git diff --cached

# 近期提交（理解上下文）
git log --oneline -5
```

**读取 spec 文档（若存在）**：

检查 `.claude/loong-claude/spec/` 目录是否存在与本次变更相关的设计文档。若存在，读取其 frontmatter 中的 `risk_level` 和 `affected_modules`，作为本次审查深度的参考依据：

```
risk_level: high   → 所有受影响模块走第一层 + 第二层完整审查
risk_level: medium → 受影响模块走第一层完整审查，第二层按需
risk_level: low    → 仅走第一层审查（正确性 + 安全性）
未找到 spec 文档   → 按变更类型判定表决定审查深度（见下方）
```

**变更类型判定**：

| 变更特征 | 类型 | 默认审查层级 |
| --- | --- | --- |
| 新增文件/函数 + 业务逻辑 | 新功能 | 第一层 + 第二层 + 第三层（兜底） |
| 修复错误消息/条件判断 | Bug 修复 | 第一层完整 + 第二层（错误处理） |
| 移动/重命名/拆分文件 | 重构 | 第一层（正确性）+ 第二层（可维护性） |
| 配置文件/依赖变更 | 配置 | 第一层（安全性）仅此 |

**排除规则**：以下文件跳过审查，在报告中标注为「已跳过」：
- 自动生成的代码（如 `src/api/` 中由 yapi-to-typescript 生成的文件）
- 锁文件（`package-lock.json`、`yarn.lock`、`pnpm-lock.yaml`）
- 第三方库代码
- 纯样式微调（仅含颜色/间距变更且无逻辑影响）

**规模预警**：若变更涉及**超过 15 个文件**，发出预警并给出选择：

> 变更范围较大（N 个文件），完整审查可能耗时较长。是否：
> a) 全量审查
> b) 仅审查核心业务文件（排除配置、类型声明、测试辅助文件）

## Step 2. 按维度审查

**审查维度按 LLM 可靠性分三层**，每层的发现在报告中以不同置信度标注。**并非每个维度都必须在每个文件中找到问题**——只记录实际发现的缺陷。

---

### 第一层：LLM 高可靠维度（必须审查，结论可直接采信）

LLM 在静态分析上有真实优势，发现的问题可直接修复，无需额外验证。

#### 1. 正确性（→ CRITICAL）

```
- 逻辑是否正确实现了意图？是否存在 off-by-one、条件反转、短路求值错误？
- 边界条件是否处理？（空数组、null/undefined、零值、极大值）
- 是否存在竞态条件或时序问题？（并发请求、事件监听器未清理）
- 数据流是否完整？（接口返回值与使用处类型匹配、状态更新后 UI 同步）
```

#### 2. 安全性（→ CRITICAL）

```
- 是否存在注入风险？（SQL 拼接、命令执行、动态 HTML 渲染）
- 用户输入是否经过校验和转义？
- 敏感数据是否暴露？（console.log 打印密钥/token、URL 参数传递敏感信息）
- 认证/授权是否缺失或可绕过？
- 是否引入了不安全的依赖？
```

#### 3. 设计（→ WARNING）

```
- 抽象层次是否恰当？（不过度设计，也不缺乏封装）
- 是否遵循 SOLID 原则？是否存在职责不清的"上帝函数"？
- 新增代码与现有架构风格一致吗？
- 是否有不必要的耦合？（直接访问跨模块内部状态、硬编码的隐式依赖）
```

#### 4. 错误处理（→ WARNING）

```
- 异步操作是否处理了失败场景？（接口报错、Promise 未 catch）
- 是否存在静默失败？（空 catch 块、吞没异常不报告）
- 错误信息是否对调试有意义？（避免笼统的"操作失败"）
- 降级策略是否合理？（加载失败是否有兜底 UI、数据异常是否有容错）
```

---

### 第二层：LLM 中可靠维度（按需审查，发现须标注「建议运行时验证」）

LLM 只能做静态推断，无运行时数据支撑。报告中此层的所有发现**必须附注**：`[静态推断，建议 profiling/运行时验证后再决定是否修复]`。

#### 5. 性能（→ WARNING）

```
- 是否存在不必要的重复计算或渲染？（缺少 memo/useMemo、循环内重复查询）
- 是否存在 N+1 查询或瀑布式请求？（可从代码结构推断，但严重程度需实测）
- 数据结构选择是否合理？（用数组做频繁查找、未考虑大数据量场景）
- 是否存在内存泄漏风险？（定时器未清理、闭包持有大对象引用）
```

#### 6. 可维护性（→ WARNING）

```
- 是否存在可提取的重复代码块？（3 行以上相同逻辑出现 2 次以上）
- 函数/文件是否过长？（单函数 > 50 行或单文件 > 300 行需关注）
- 公共接口变更是否更新了相关文档和类型定义？
- 是否有未使用的变量、导入或死代码？
```

---

### 第三层：工具链优先维度（LLM 仅做兜底，建议交给 lint/formatter）

此层问题应由 ESLint/Prettier/tsc 等工具链在 CI 阶段捕获。LLM 仅在发现工具链**明显遗漏**的问题时记录，否则跳过，不在报告中列出 INFO 条目。

#### 7. 风格与命名（→ INFO，仅记录工具链无法捕获的问题）

```
- 变量/函数命名是否描述性足够？（避免 d、tmp、data 等泛化名称）——工具链无法检测，LLM 审查
- 代码风格是否与项目一致？（引号、缩进、分号）——应由 Prettier 覆盖，LLM 跳过
- 是否有应删除的注释掉的代码或调试日志？——工具链可配置规则，LLM 兜底
- 注释是否解释"为什么"而非"做了什么"？——工具链无法检测，LLM 审查
```

## Step 3. 生成审查报告

### 严重等级定义

| 等级 | 含义 | 是否阻塞提交 |
| --- | --- | --- |
| **CRITICAL** | 安全漏洞、逻辑错误、数据丢失风险 | **是**（必须修复） |
| **WARNING** | 设计缺陷、性能隐患、可维护性问题 | 否（建议修复） |
| **INFO** | 工具链未覆盖的命名/注释问题 | 否（可选改进） |

### 报告格式

```
## Code Review Report

**变更范围**: N files, +X -Y
**变更类型**: 新功能 / Bug 修复 / 重构 / 配置
**审查深度**: 第一层（正确性/安全性/设计/错误处理）[+ 第二层（性能/可维护性）] [+ 第三层兜底]
**参考依据**: spec 文档 <路径> (risk_level: high/medium/low) / 无 spec，按变更类型判定
**整体评估**: ✅ 通过 / ⚠️ 有条件通过 / ❌ 不通过

### CRITICAL（必须修复，阻塞提交）
1. [维度] 文件路径:行号
   - 问题: 具体描述
   - 建议: 修复方案

### WARNING（建议改进）
2. [维度] 文件路径:行号
   - 问题: 具体描述
   - 建议: 改进方案
   - 置信度: 直接修复 / 静态推断，建议运行时验证后再决定

### INFO（可选优化，工具链未覆盖）
3. [维度] 文件路径:行号
   - 问题: 具体描述
   - 建议: 优化建议

### 审查摘要
- 已跳过: [文件列表及原因]
- 已审查: N 个文件，审查层级：[第一层 / 第一层+第二层 / 完整]
- 发现: N 个 CRITICAL / N 个 WARNING（其中 N 个需运行时验证）/ N 个 INFO
```

### 整体评估判定

```
存在 CRITICAL         → ❌ 不通过
仅存在 WARNING (≤3)   → ⚠️ 有条件通过
仅存在 WARNING (>3)   → ⚠️ 有条件通过（建议优先处理前 3 项）
仅存在 INFO 或无发现  → ✅ 通过
```

## Step 4. 用户决策

展示报告后，等待用户选择：

> 审查完成。请选择下一步：
> a) **修复 CRITICAL** — 仅修复阻塞提交的问题
> b) **修复 CRITICAL + WARNING** — 修复所有重要问题（跳过标注「需运行时验证」的条目）
> c) **全部修复** — 修复所有发现的问题
> d) **标记并继续** — 记录为技术债，直接进入提交流程
> e) **跳过** — 忽略审查结果，直接进入提交流程

**选择 a/b/c 后**：使用 `loong-claude:debugging` skill进行具体问题排查并执行修复，进入 Step 5 的**有界修复循环**。

**选择 d 后**：生成技术债记录，附于 commit message 的 body 中（非必须）。

---

## Step 5. 有界修复循环

### 修复原则

- **最小侵入**：只修复报告中具体标注的问题，不顺手重构周围代码。修改范围不得超出报告中标注的文件和行号范围。
- **逐项修复**：按 CRITICAL → WARNING → INFO 顺序，逐项修复并验证。
- **单轮单改**：每次只修复一个问题，避免引入连锁错误。

### 修复后验证

```bash
# 针对被修复文件运行相关测试（优先，而非全量）
npm test -- --testPathPattern=<被修复文件路径>

# 类型检查
npx tsc --noEmit

# 确认修改范围未超出授权文件
git diff --stat
```

### 重审规则（有界，防止无限递归）

修复完成后，执行一次轻量重审（仅检查被修改的文件，不重跑全量审查）：

```
重审发现的新问题数 < 本轮修复的问题数？
  是 → 继续，进入下一轮修复（若还有剩余问题）
  否 → 停止修复，回滚本轮变更，输出：
       「本轮修复引入了等量或更多新问题，建议人工介入处理。
         已回滚修复，原始问题仍在报告中。」

已完成第 2 次重审？
  是 → 无论结果如何，终止循环，输出：
       「已完成 2 轮修复迭代，剩余问题建议人工处理或记录为技术债。」
       列出剩余未修复的条目，等待用户决策。
  否 → 继续
```

**上限说明**：最多 2 轮重审（初次修复 + 1 次迭代），超出后终止并移交用户。这防止了 LLM 在「风格修复 → 引入新风格偏差 → 再次修复」的循环中无法收敛。

---

## **禁止**的反模式

- 未经用户授权自行修复发现的问题
- 审查时夹带与报告无关的重构建议（"顺手改进"）
- 对自动生成的代码进行审查（浪费时间）
- 所有发现都标为 CRITICAL（丧失分级意义）
- 只报告问题不给具体修复建议
- 跳过上下文收集直接开始逐文件扫描（缺乏整体理解会导致误报）
- 审查报告中使用模糊描述（"这段代码不好"而非具体的缺陷描述）
- 对第二层（性能/内存）的静态推断结论不加标注，以确定性语气输出（会破坏用户对报告的信任）
